# Infrastructure as Code -- Snowflake

Terraform manages the Snowflake surface for this project's Snowflake retarget, run against the real, live account `DZVUEJF-DF04786` (org `DZVUEJF`, account `DF04786`, AWS-hosted). This is a **parallel stack**, not a replacement for `infra/` -- the Databricks stack keeps running unchanged so the two platforms can be compared before any cutover decision. See `docs/snowflake_migration_plan.md` for the full migration plan and parallel-run/cutover checklist.

## Why a separate directory instead of a second provider in `infra/`

A dedicated `infra_snowflake/` directory with its own Terraform state means a mistake here (a bad `plan`, an accidental `destroy`) can never touch the live, working Databricks resources in `infra/`. The two stacks share no state, no provider config, and no resource addresses.

## What's code-managed vs. manual

| Managed by Terraform | Managed manually / by dbt |
|---|---|
| Database (`snowflake_database`) | Table/view DDL and data (owned by dbt -- see `transform/`) |
| Schemas: BRONZE/SILVER/GOLD/OPS (`snowflake_schema`) | Row data (owned by `data_generation/` and `ingestion/`) |
| Warehouses: `TRANSFORM_WH`, `BI_WH` (`snowflake_warehouse`) | Snowflake account signup/trial (one-time, human) |
| RBAC roles (`snowflake_account_role`) and role grants (`snowflake_grant_account_role`) | Admin user's key-pair registration (`ALTER USER ... SET RSA_PUBLIC_KEY`, run once in Snowsight -- see credential handoff notes) |
| Grants (`snowflake_grant_privileges_to_account_role`) | |
| Bronze landing stage + shared Parquet file format (`stage.tf`) | Actually PUTting/COPYing files -- see `scripts/load_bronze_to_snowflake.py` |
| Account-wide `QUOTED_IDENTIFIERS_IGNORE_CASE` (`account_parameters.tf`) | |
| CI/CD service user (`snowflake_service_user.ci_cd`) | GitHub Actions secret containing its private key (never generated by or visible to this Terraform config) |

Same deliberate split as `infra/`: Terraform owns the *access surface*, dbt owns the *data*.

## Credentials

Terraform authenticates as the account owner (`MITCHELLMC`, `ACCOUNTADMIN`) via key-pair auth. Private keys live outside the repo at `~/.snowflake/keys/` (`mitchellmc_rsa_key.p8` for the admin bootstrap identity, `recon_ci_svc_rsa_key.p8` for the CI service user -- the latter's private key is what actually goes into a GitHub Actions secret; Terraform only ever touches its **public** key, embedded via `local.ci_cd_public_key` in `service_user.tf`). Nothing under `~/.snowflake/` is tracked by this repo.

## Bootstrap ordering (first-time apply only)

`grants.tf`'s five named-object grants (`FCT_DAILY_CASH_POSITION` x2, `FCT_FUNDING_COST_SUMMARY` x2, `VW_EXCEPTION_QUEUE_MASKED`) reference Gold objects that don't exist until dbt has built them at least once -- identical constraint to `infra/grants.tf`'s table-level grants. Verified end-to-end on this real account:

```bash
cd infra_snowflake
terraform init
terraform apply -var-file=environments/dev.tfvars \
  -target=snowflake_database.this \
  -target=snowflake_schema.bronze -target=snowflake_schema.silver \
  -target=snowflake_schema.gold -target=snowflake_schema.ops \
  -target=snowflake_warehouse.transform -target=snowflake_warehouse.bi \
  -target=snowflake_account_role.engineering -target=snowflake_account_role.treasury_viewers \
  -target=snowflake_account_role.finance_analysts -target=snowflake_account_role.bi_consumers \
  -target=snowflake_grant_account_role.engineering_to_sysadmin \
  -target=snowflake_grant_account_role.treasury_viewers_to_sysadmin \
  -target=snowflake_grant_account_role.finance_analysts_to_sysadmin \
  -target=snowflake_grant_account_role.bi_consumers_to_sysadmin \
  -target=snowflake_grant_account_role.engineering_to_admin_user \
  -target=snowflake_service_user.ci_cd \
  -target=snowflake_grant_account_role.engineering_to_ci_cd

# full apply -- creates the stage/file format/account parameter and
# every grant EXCEPT the five named-object ones (those still fail: the
# Gold tables don't exist yet). This step's errors are expected, not a
# sign something's wrong.
terraform apply -var-file=environments/dev.tfvars

python scripts/load_bronze_to_snowflake.py          # from repo root
cd transform && dbt deps && dbt seed --target snowflake && \
  dbt build --target snowflake                       # creates the Gold tables

cd ../infra_snowflake
terraform apply -var-file=environments/dev.tfvars    # now the 5 named-object grants succeed too
```

After this first bootstrap, steady-state applies don't need `-target` -- the named objects already exist.

## Environments

Only `dev` (`environments/dev.tfvars`) is actually applied, same posture as `infra/`. `environments/prod.tfvars` documents a second environment's settings without provisioning it. **Do not** run `terraform apply -var-file=environments/prod.tfvars` against this directory -- it shares dev's state, so it would replace (destroy + recreate) the dev database rather than create a parallel one. Read `terraform plan` output for "forces replacement" first, every time, same lesson `infra/README.md` documents for the Databricks stack.

## Known differences from the Databricks stack

- **Case folding**: Snowflake uppercases unquoted identifiers, so this stack's database/schema/object names are declared in uppercase (`MERCHANT_RECON_PROJECT_DEV`, `BRONZE`, `FCT_DAILY_CASH_POSITION`, ...) where the Databricks catalog used lowercase. Purely cosmetic, not a behavior difference.
- **No `storage_root` drift gotcha** -- Snowflake databases have no metastore-assigned attribute equivalent, so no `lifecycle { ignore_changes }` block is needed anywhere in this stack.
- **Schema-level SELECT needs two grants, not one** -- Databricks' schema-level `SELECT` grant cascades to every table in the schema automatically. Snowflake requires an `ALL` grant (existing objects) plus a separate `FUTURE` grant (objects dbt creates later) to get the same effect -- see the `finance_analysts_*_select_existing` / `_future` resource pairs in `grants.tf`.
- **Warehouse USAGE is explicit** -- Databricks SQL-warehouse access wasn't Terraform-managed in `infra/`; Snowflake roles get nothing without an explicit `USAGE` (and `OPERATE`, for the engineering role) grant on a warehouse, added at the top of `grants.tf`.
- **No `jobs.tf` equivalent** -- Snowflake Terraform has no resource for running dbt on a schedule. Scheduling moved to a GitHub Actions workflow (`.github/workflows/snowflake_daily.yml`) instead of a warehouse-side job resource.
- **`QUOTED_IDENTIFIERS_IGNORE_CASE` set account-wide** (`account_parameters.tf`) -- has no Databricks equivalent or need. `scripts/load_bronze_to_snowflake.py`'s `INFER_SCHEMA`-driven Bronze table creation preserves the Parquet files' exact lowercase column-name case as case-sensitive quoted identifiers; dbt's generated SQL references columns unquoted (case-folded uppercase). Confirmed by direct testing that this must be an account-level (not per-session) setting -- it's a session-behavior parameter that every future session (local dbt CLI, `RECON_CI_SVC` in CI) needs to inherit, not a stored table property fixed at creation time.
- **Known non-convergent `terraform plan` drift on `snowflake_stage.bronze_landing`** -- the provider re-escapes the `file_format` attribute's embedded quote characters differently on every state refresh than how the config declares them, so `plan` always shows a cosmetic 1-resource diff on this stage even right after a clean apply. Tried and rejected: an unqualified format name avoids the string-escaping issue but Snowflake's `ALTER STAGE` rejects it outright (confirmed by testing). Functionally harmless either way -- `PUT`/`COPY INTO` both work -- left as a known, accepted limitation rather than chased further, same spirit as `infra/README.md`'s documented gotchas.
